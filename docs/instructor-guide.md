# 講師向け運用メモ — Claude スキルアップ講座

**対象**：本研修を進行する講師・ファシリテーター
**目的**：研修を実施しやすくするための、進行・運用に直結する内部メモ
**位置づけ**：受講者向けページではない。受講者には共有しないこと
**最終更新**：2026-05-01

---

## 1. 研修全体の位置づけ

### 対象者
- ChatGPT / Gemini などの生成AIを使ったことがある
- Claude は初めて、またはほとんど使ったことがない
- ビジネス職、コンサル、企画、リサーチ、資料作成担当
- 業務の初速・成果物作成・品質と再現性を Claude で上げたい

### 研修の目的
受講者が次の3点を実務で実行できる状態になること：
1. **業務の初速を上げる**（Chat で壁打ち・下書き・要約・論点整理）
2. **成果物を作る**（Office / Research / Artifact / Cowork で文書・スライド・調査・プロトタイプ）
3. **品質と再現性を上げる**（Skills / MCP / Claude Code / CLAUDE.md / チーム展開）

### 全2回の流れ
- **第1回**：Claudeで仕事の初速を上げ、成果物を作る（14セクション：前半7＋後半7）
  - 前半（1-1〜1-7）：Claude の特徴理解 → Chat → Project / Artifact
  - 後半（1-8〜1-14）：成果物の幅 → Office → Research → Artifact プロトタイプ → Chrome → Cowork → Scheduled tasks
- **第2回**：品質と再現性を上げる（5セクション：2-1〜2-5）
- **修了テスト**：32問・全回からシャッフル出題

### 研修モードと自習モードの使い分け
| モード | 用途 | UI |
|---|---|---|
| **研修モード** | 講師投影用。スライド型で1セクション=1画面 | カバー → セクションスライド × N |
| **自習モード** | 受講者の復習・予習・参照用 | Webサイト型。サイドバー目次・前後ナビ |

研修中は研修モードを投影、休憩・終了後・宿題で自習モードを案内する。

### TIPS集の位置づけ
- 自習モードの**補助教材**
- 「全部覚えるもの」ではなく「業務で必要になったときに参照するもの」
- カテゴリ別（8カテゴリ）、レベル別（基本/中級/上級）、キーワード検索でフィルタ可能
- 上級・補助 TIPS（権限モード、`.claude` 操作、自動化）は**必須ではない**
- 学習セクションから関連 TIPS へ、TIPS から関連セクションへ相互リンクあり

## 3. 第1回の講師メモ

### この回のゴール
- 受講者が Claude の全体像（作業スタイル × 利用場所）を頭に入れる
- Chat の基本操作（壁打ち・下書き・要約）が手元でできる
- Project / Artifact で成果物化の入口を体験する
- 「出力は初稿、責任は人間」を腹落ちさせる

### 講師が強調すべきメッセージ
1. **「ChatGPT / Gemini を使ったことがある人前提」**：冒頭で挙手確認。基本操作（チャット・添付・Web検索）はそのまま活かせる旨を伝える
2. **「3つの作業スタイル × 利用場所」を分けて説明**：
   - 作業スタイル：Chat / Cowork / Claude Code（**何を任せるか**）
   - 利用場所：Web / Desktop / Mobile / CLI / IDE（Cursor / VS Code / JetBrains 等）（**どこで使うか**）
   - 「2軸を混ぜると混乱する」とはっきり言うのがコツ
3. **「Claude Code は必要に応じて使う高度な選択肢」**：初心者はまず Web 版／Claude Desktop で Chat / Cowork を使えば十分、と先に伝える。Claude Code を IDE で使う場合は **社内に Cursor の活用ノウハウがある**が **Cursor は全員必須ではない**。CLI はエンジニアや慣れた人向け補足。**「Cursor を全員に推奨している」「CLI を覚えないと使えない」のどちらの誤解も招かない**ように説明する
4. **「迷ったら Chat から」**：Cowork は後半（1-13）、Claude Code は第2回（2-3）で詳しく扱うため、前半では深入りしない
5. **「Claude の出力は必ず人間が確認する」**：1-6「出力を疑う・直す・責任を持つ」のセクションでしっかり時間を取る

### デモするとよい箇所（前半）
- **1-1〜1-2 通しでスライド読み上げ**（Claude の3つの強み・機能マップ）
- **1-3 の2つのテーブル**（作業スタイル × 利用場所）：投影しながら指でなぞって整理
- **1-4「Chat の3つのパターン」**：壁打ち・下書き・要約のうち1つを実演
- **1-5「良い指示の出し方（3要素）」**：Before / After の比較を見せる
- **1-7「Project / Artifact」**：実際に Project を作る画面を見せる

### 演習に使うとよい箇所（前半）
- **1-4 の壁打ち or 下書き**：自分の業務テーマで1件、Chat で壁打ちする
- **1-7 ハンズオン**：自分の案件用 Project を1つ作ってみる

### 飛ばしてよい補足
- 1-2 の「品質・連携を支える機能」（Skills / MCP）：第2回で詳しく扱うため、前半では「こういうものがある」程度
- 1-3 の利用場所テーブルの IDE 行：受講者層に応じて軽く触れる程度（IDE/CLI は必要になってから）
- 1-2 の Office アドイン・Chrome 拡張：後半（1-9 / 1-12）で扱う

### 想定される質問（前半）
- 「ChatGPT と何が違うんですか？」 → 1-1 / 1-2 を参照（長文に強い、指示に忠実、考えるパートナー）
- 「Claude Code は CLI 必須ですか？」 → 1-3 の info-box を読み返す（CLI でも IDE でも使える、初心者はまず Desktop の Chat / Cowork で十分）
- 「Project と Artifact の違いは？」 → 1-7 の比較図（Project = 案件の前提保持／Artifact = 成果物生成）

### 第1回のキーフレーズ
> 「Claude は<strong>会話相手</strong>であり<strong>業務の相棒</strong>。検索の代わりではなく、考える伴走者として使う」

## 4. 第1回 後半の講師メモ（1-8〜1-14：成果物を作る）

### この回のゴール
- 受講者が「Claude で作れる成果物の幅」を体感する
- Office / Research / Artifact / Chrome / Cowork / Scheduled tasks をそれぞれ役割で理解する
- スライド・提案書・調査の段階生成の型を持ち帰る

### 講師が強調すべきメッセージ
1. **「成果物作成は段階的に」**：いきなり完成形を作らせない。構成 → メッセージ → 図表案 → 文言 → デザインの順
2. **「プロトタイプ ≠ 本番システム」**：Artifact のプロトタイプは**議論のたたき台**。クライアントに見せるときは必ず「試作品です」と伝える
3. **「Research は示唆まで変換する」**：情報収集で終わらせず「機会・脅威・推奨アクション」まで Claude に出させる
4. **「Cowork は作業コピーで使う」**：元ファイルは触らせない。重要・機密ファイルはバックアップ必須
5. **「プロトタイプ作成は主軸ではない」**：「プロトタイプ」の一例として扱う。提案書やスライドのほうがより一般的に使う

### デモするとよい箇所
- **1-9 のスライド段階生成**：構成→メッセージ→図表案→レビュー観点の4ステップを画面で読み上げ
- **1-10 の Research → 示唆変換**：Before/After（情報収集で止まる vs 示唆まで変換する）
- **1-11 の Artifact 試作品**：診断ツール or シミュレーターを実際にClaudeに作らせる
- **1-13 の Cowork**：作業用フォルダを切ってファイル作業を任せる流れ

### 演習に使うとよい箇所
- **1-9**：自分の業務で1枚スライドの構成案を Claude で作る
- **1-10**：自分の関心テーマで Research を1件回し、示唆まで変換させる
- **1-11**：簡単な診断ツール（5問程度）の Artifact を1つ作る

### 飛ばしてよい補足
- 1-12 Chrome 拡張：使わない受講者には軽く紹介のみ
- 1-14 Scheduled tasks：個人で使うか不確かな場合は概念紹介のみ

### 想定される質問
- 「Office アドインと Cowork は何が違う？」 → Office アドインは Excel/PPT/Word **の中で**Claude を呼ぶ／Cowork は**ファイルを Claude に渡して**任せる
- 「Research の出典は信頼できる？」 → 出典 URL は無効なケースもある。重要な数字は必ず一次情報で確認
- 「プロトタイプはクライアント提出 OK？」 → 「議論用の試作品」と明示すれば OK。本番システムではないことを伝える

### 第1回 後半のキーフレーズ
> 「Claude は<strong>初稿を作る助手</strong>。完成形に磨くのは人。<strong>段階を分けて作らせる</strong>のが品質の鍵」

## 5. 第2回の講師メモ

### この回のゴール
- 受講者が「個人の便利技」を「チームで再現できる仕組み」に変える視点を持つ
- Skills / MCP / Claude Code / CLAUDE.md の役割が分かる
- チーム展開の運用ルール（オンボーディング・レビュー・情報管理）を設計できる

### 講師が強調すべきメッセージ
1. **「Skills = 繰り返し業務の標準化」**：1回しか使わない作業は Skills 化しない。繰り返し × 手順化可能 × 品質基準あり
2. **「MCP / Connectors = 情報接続」**：Slack / Notion / Drive 等を Claude につなぐ。アクセス権を持つ範囲のみ参照可能
3. **「Claude Code = 高度な作業を任せる」**：必要になったときに CLI または IDE で使う高度な選択肢。社内には Cursor の活用ノウハウがあるが Cursor は全員必須ではない。CLI はエンジニア・慣れた人向け補足
4. **「CLAUDE.md = 品質基準書」**：Hooks / Subagents は品質を固定する仕組み。最初は CLAUDE.md だけで十分
5. **「TIPS は全部覚えない」**：2-3 の TIPS は info-box で「業務で必要になったら参照」と明示している。受講者に圧をかけない
6. **「権限モード・自動化は補助扱い」**：Bypass Permissions、`.claude` フォルダ操作などは上級者向け。チームルール前提
7. **「チーム展開はオンボーディング資料・レビュー観点・禁止事項・テンプレートを揃える」**

### デモするとよい箇所
- **2-1 の Skills 構成要素4ステップ**：タスク定義 → 手順 → ルール → 出力形式
- **2-3 の準備5点**：作業フォルダ・目的・成果物・禁止事項・変更対象（始める前のチェックリストとして強調）
- **2-3 の「うまく動かない時の戻り道」**：強い指示を出すより準備に戻る
- **2-4 の CLAUDE.md / DESIGN.md / CONTENT.md / REVIEW.md** の役割分担
- **2-5 の新メンバーオンボーディング資料**

### 演習に使うとよい箇所
- **2-1**：自分の業務から Skills 化候補を3つ洗い出す
- **2-3**：Claude Code への安全な修正指示テンプレートを書く
- **2-5**：チーム共通の禁止事項を5項目書く

### 飛ばしてよい補足
- Hooks / Subagents の詳細実装：「最初は CLAUDE.md だけで十分」と伝え、深入りしない
- TIPS集の上級・補助 TIPS（権限モード・`.claude` 操作・自動化など）：必要な人が後から参照する位置づけ

### 想定される質問
- 「Skills は何個作るのが良い？」 → 数より質。週1回以上使う業務 3〜5 個から始める
- 「CLAUDE.md は誰が管理する？」 → チームでオーナーを1人決め、レビュー前提で更新する
- 「Cursor 以外でも Claude Code 使える？」 → CLI も VS Code / JetBrains 等の IDE 拡張も使える。Cursor は社内に活用ノウハウがあるという紹介で、全員必須ではない
- 「権限モードは使うべき？」 → 業務利用では原則使わない。作業コピーかつチームルール前提でのみ

### 第2回のキーフレーズ
> 「<strong>個人の便利技を、チームの標準に。標準を、組織の力に。</strong>」（2-5 の big-message そのまま）

## 6. 演習運用メモ

### 自分の業務テーマを使わせる場合の注意
- **必ず冒頭で機密情報の扱いを確認**：演習で Claude に渡してよい情報の範囲をクライアントの方針に合わせて明確化
- **クライアント名・契約金額・個人情報は入力させない**：匿名化・サマリー化を促す
- **「演習は使い方の体験。完成度ではない」**：上手く出せなくても OK と先に伝えると参加率が上がる

### サンプルテーマを用意する（業務テーマがない人向け）
| テーマ | 適した回 | 概要 |
|---|---|---|
| 架空の DX 推進プロジェクトの提案書構成 | 第1回 後半 | 業界 X 社向け、3年計画、5枚スライド |
| 業界動向リサーチ（生成AI の B2B 導入事例） | 第1回 後半 | Research で5件抽出 → 示唆3点 |
| 議事録から決定事項抽出 Skill | 第2回 | 架空の会議メモを渡して整理 |
| チーム共通 CLAUDE.md の禁止事項5項目 | 第2回 | 自社向けに具体化 |

### 発表・共有の進め方
- 演習後の共有時間は**1人 2 分以内**を目安
- 「うまくいったプロンプト」「うまくいかなかった点」「次に試したいこと」の3点で発表
- 全員発表が時間的に厳しい場合は、ペアワーク → 代表数名のみ発表
- 共有された成功例はチーム共通テンプレート化の候補としてその場で記録する

### 演習は完成度より体験を重視
- 1回の演習で完璧な成果物を求めない
- **「指示の型」が体験できればゴール**（範囲限定・禁止事項・段階生成・確認）
- 「うまくいかなかった例」のほうが学びが多いと事前に伝える

## 7. よくある質問への講師回答例

### Q1. ChatGPT / Gemini と何が違うのですか？
**A.** 基本操作は共通です。違いは設計思想と得意領域。Claude は<strong>長文に強い（200K トークン）・指示に忠実・考えるパートナーとして設計</strong>されています。たとえば長い議事録を丸ごと渡して構造化させたり、「箇条書き3点で・です・ます調で」のような形式制約を守らせるのに向いています。「ChatGPT で身につけたスキルはそのまま活かせます」と安心させてください。

### Q2. まず何から使えばよいですか？
**A.** **Chat から始めてください。**自分の業務で「ちょっと相談したい」「下書きが欲しい」「長文を要約したい」場面で使うのが最も早く効果が出ます。Cowork は第1回 後半、Claude Code は第2回で扱いますが、まずは Chat の壁打ち・下書き・要約の3パターンを試すのがおすすめです。

### Q3. Claude Code は CLI を覚えないと使えませんか？
**A.** いいえ。**CLI を覚えなくても Claude Code は使えます。** Claude Code は IDE（Cursor / VS Code / JetBrains 等）でも CLI でも利用可能です。ただし Claude Code は「高度な作業を任せる」ときの選択肢で、**初心者はまず Web 版または Claude Desktop の Chat / Cowork で十分**な場面が多いです。社内には Cursor の活用ノウハウもありますが、Cursor は全員必須ではありません。CLI はエンジニアや慣れた人向けの補足として扱います。

### Q4. Cursor と Claude Code の関係は何ですか？
**A.** Cursor は IDE（コードを書く環境）、Claude Code は AI による作業エージェント機能です。Claude Code は CLI でも、Cursor / VS Code / JetBrains などの IDE でも利用できます。**社内には Cursor の活用ノウハウもあるので IDE 利用時の選択肢のひとつ**ですが、**Cursor は全員必須ではありません**。Claude Code 自体、初心者は無理に使わず、Web 版／Claude Desktop の Chat / Cowork から始めて、複数ファイル編集やアプリ改修が必要になったタイミングで触れるので十分です。

### Q5. Cowork と Claude Code はどう使い分けますか？
**A.** **Cowork は手元のファイル群を渡して成果物化する用途**（PPTX / XLSX / DOCX を直接出力）。**Claude Code はプロジェクトフォルダ単位で複数ファイルを横断する高度な作業**（コード・アプリ・データ処理・自動化）に向きます。提案書1本作るなら Cowork、社内ツールを作るなら Claude Code、と覚えてください。

### Q6. Claude の出力はそのまま使ってよいですか？
**A.** **そのまま使わないでください。**どの機能で作った成果物も「優秀な同僚の初稿」です。特に Research の数値・出典、Artifact の計算ロジック、Office 連携の分析結果、コードや自動化スクリプトは、自分で確認してから使ってください。**最終責任は常に人間にあります。**

### Q7. プロトタイプはクライアントに見せてもよいですか？
**A.** **「議論用の試作品です」と明示すれば見せて問題ありません。** Artifact で作るものは本番システムではなく、論点を具体化するためのたたき台です。データの保存機能やセキュリティ対策はないことを必ず伝え、本番開発は別途必要であることを明確にしてください。

### Q8. Skills はどんな業務に向いていますか？
**A.** 次の3条件が揃ったら Skills 化候補です：(1) 繰り返し発生する、(2) 手順が明確化できる、(3) 品質基準がある。具体例は議事録、提案書レビュー、調査レポート、PPT 構成生成、Excel 分析、UI レビュー、プロトタイプ手順。一度しか使わない作業は Skills 化しなくて OK です。

### Q9. 機密情報はどう扱えばよいですか？
**A.** チームで「何を Claude に渡してよいか」をルール化してください。**個人情報・契約金額・NDA 対象情報は慎重に扱い、迷ったら上長に確認**するルールにしておくと安全です。匿名化・サマリー化した上で使うのが基本です。インシデント発生時の対応フローも事前に決めておきましょう。
